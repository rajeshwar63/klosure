# Step 04 — Buyer view schema

**Sprint:** B
**Goal:** Define the structured shape of `klo_state.buyer_view` — the data blob that drives the entire buyer dashboard. This is the contract between the LLM (which writes it in step 05/06) and the UI (which reads it in step 07).

## Files

- `supabase/functions/_shared/klo-state-types.ts` — extend `KloState` with `buyer_view`
- `supabase/functions/klo-respond/index.ts` — extend `KLO_OUTPUT_TOOL` schema (or create a separate buyer-view tool — see step 06)

## Why JSONB inside klo_state, not a separate table

`klo_state` is already a JSONB blob on `deals`. Adding `buyer_view` as a nested key keeps everything consistent:
- One realtime channel (already enabled on `deals`)
- One RLS policy (already covers buyer access via shared mode)
- One read for the UI (no joins, no extra query)
- Same provenance + history mechanics

A separate table would require its own RLS, its own realtime publication, and a join — all for no benefit. The buyer view is fully derived from the chat; no need to normalize it.

## Type additions

In `supabase/functions/_shared/klo-state-types.ts`, add these types:

```typescript
// =============================================================================
// Phase 8 — Buyer view structured shape
// =============================================================================

export type SignalLevel = 'strong' | 'mixed' | 'weak'

export interface BuyerSignal {
  // Three buyer-facing health indicators that sit under the Klo brief.
  // These are NOT the seller's confidence factors. Buyer-relevant only.
  kind: 'timeline_health' | 'stakeholder_alignment' | 'vendor_responsiveness'
  level: SignalLevel
  one_line_why: string  // ≤ 14 words explaining the level
}

export interface BuyerPlaybookItem {
  action: string             // imperative, ≤ 12 words. "Loop in your CISO before Friday"
  why_it_matters: string     // 1 sentence, Klo's voice
  who: 'you' | 'your_team' | 'vendor' | string  // free text — "your CFO", "your CISO", "vendor's SE"
  deadline: string | null    // ISO date or null
  status: 'not_started' | 'in_flight' | 'done'  // server-side default 'not_started'
  source_message_id: string | null
}

export interface BuyerStakeholderTake {
  // Buyer-side internal stakeholders only — vendor team is captured separately.
  name: string
  role: string                 // "CFO", "CISO", "Procurement Lead", "End-user lead"
  engagement: 'aligned' | 'engaged' | 'quiet' | 'blocker' | 'unknown'
  klo_note: string | null      // 1 sentence — what to do about this stakeholder
}

export interface BuyerRisk {
  label: string                // ≤ 10 words. "Procurement timeline"
  why_it_matters: string       // 1-2 sentences in Klo's voice, buyer-side framing
  mitigation: string           // 1 sentence — what the buyer should do
}

export interface BuyerRecentMoment {
  // Buyer-friendly history feed. Last 3-5 important things from the seller's chat.
  date: string                 // ISO date
  text: string                 // ≤ 16 words. "Vendor sent SOC 2 report"
}

export interface BuyerView {
  // The buyer-facing brief — the hero card on the dashboard.
  // Written TO the buyer, not about them. 3-5 sentences. Klo's voice.
  // Action-oriented framing. Never reveals seller strategy or confidence.
  klo_brief_for_buyer: string

  // Three signals shown under the brief. Always 3 — one of each kind.
  signals: BuyerSignal[]

  // 3-5 specific moves the buyer should make this week.
  playbook: BuyerPlaybookItem[]

  // Buyer's internal stakeholder map. 3-8 people max.
  stakeholder_takes: BuyerStakeholderTake[]

  // 2-3 risks Klo is watching, framed as opportunities to act.
  risks_klo_is_watching: BuyerRisk[]

  // Single number 0-100 — buyer-facing "deal momentum" signal.
  // NOT the seller's confidence score — different framing, different scale meaning.
  // High = deal is moving forward (commitments kept, stakeholders engaged).
  // Low = deal is stalling (commitments slipping, stakeholders quiet).
  momentum_score: number | null

  // 'up' | 'down' | 'flat' — direction of momentum vs last update.
  momentum_trend: 'up' | 'down' | 'flat' | null

  // 3-5 buyer-friendly history items, oldest to newest.
  recent_moments: BuyerRecentMoment[]

  // Bookkeeping
  generated_at: string         // ISO timestamp — when this buyer_view was last written
  generation_reason: 'initial' | 'material_change' | 'manual_refresh'
}

export interface KloState {
  // ... all existing fields unchanged ...

  // NEW — Phase 8
  buyer_view?: BuyerView | null
}
```

## Tool schema extension

In `supabase/functions/klo-respond/index.ts`, the existing `KLO_OUTPUT_TOOL` does NOT need to be extended. The buyer view is generated by a **separate** LLM call (step 06), not the main extraction call. This is a deliberate choice:

1. Main extraction stays cheap and fast (output tokens unchanged)
2. Buyer view extraction is gated (doesn't run on every turn)
3. Quality concerns isolated — if buyer-view prompt has issues, main coaching is unaffected

So the main tool schema stays as-is. The buyer-view-specific tool schema lives in a new file (defined in step 06).

## Migration of existing klo_state rows

Existing rows have `klo_state` without a `buyer_view` key. That's fine — the type marks it optional (`buyer_view?: BuyerView | null`). No migration needed. The first time the buyer-view extraction runs on a deal, it populates the field.

UI (step 07) must handle the case where `buyer_view` is missing — render an empty state: *"Klo hasn't written your buyer dashboard yet. Send a few more messages and we'll generate it."*

## RLS

No changes needed. `klo_state` is already covered by the existing policies on `deals`:
- Seller reads/writes their own deals
- Buyer reads via shared mode (`deals.mode = 'shared'`)
- Manager reads via team membership

The buyer view inherits these — buyer reads `klo_state.buyer_view` because they can already read `klo_state` on shared deals.

**Important seller-vs-buyer split inside the JSONB**: existing fields like `klo_take_seller`, `confidence`, `factors_dragging_down` are still in `klo_state` and a buyer can technically read them via JSONB access. This was true before Phase 8 — the UI is the gatekeeper, never showing seller-only fields to buyers. Phase 8 doesn't change this; it's already the architecture.

If you want defense-in-depth on this, that's a future phase — not Phase 8. Out of scope here.

## Realtime

The existing realtime publication on `deals` already carries `klo_state` updates. When the buyer-view extraction writes back, the buyer dashboard re-renders automatically. No new wiring needed.

## What this step does NOT do

- Does NOT write any data — that's step 06
- Does NOT change the main `KLO_OUTPUT_TOOL` schema — buyer view is a separate call
- Does NOT add SQL — the JSONB shape is enforced by TypeScript types and the LLM tool schema, not by Postgres
